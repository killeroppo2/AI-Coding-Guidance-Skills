# Self-evolving AI development kernel

A kernel framework for autonomous AI agent execution: BOOT.md takes over the agent's context, a YAML-defined workflow graph drives iteration through plan/code/test/review/reflect/evolve nodes, and the evolution engine lets the kernel modify its own prompts and graph while constitution.md holds three files immutable. The runner CLI provides the entry point with dry-run support, while knowledge/ and memory/ directories give the stateless agent filesystem-based persistence.

**Watch for:** The evolution engine's immutability guard is bypassable via path normalization tricks (confirmed). The runner loop always picks the first transition regardless of actual conditions, making non-dry-run execution deterministic but semantically incorrect (confirmed). BOOT.md instructs the agent to update state.yaml directly, which conflicts with the runner's own state management (likely).

## High-level view

The kernel adopts a "files are memory" architecture where an AI agent starts each iteration from scratch, reads BOOT.md to acquire its role, then follows a 7-node workflow graph. It never imports an LLM SDK or assumes a particular model; the runner reads prompts and would hand them to whatever agent runtime invokes it. The trade-off is that the runner itself doesn't actually call an LLM, so in non-dry-run mode it traverses the graph mechanically, always taking the first available transition.

The evolution engine validates proposed changes against a frozenset of protected paths before writing to disk, and logs all changes to a JSONL history with rollback support. The guard is correct for exact-match paths but doesn't normalize or canonicalize, which means alternate representations of the same file (`./kernel/BOOT.md`, `kernel//BOOT.md`) would pass validation.

The test suite is thorough: 171 tests at 99% coverage, with a dedicated file (`test_coverage_gaps.py`) that systematically targets uncovered branches. Tests exercise real code paths (file I/O, YAML serialization, graph mutations) rather than mocking everything away. The main gap is that no test verifies the immutability bypass scenario or validates evolution with adversarial path inputs.

1. **Immutability bypass via path variants** — `validate_change` compares against exact strings in `IMMUTABLE_FILES` but doesn't normalize paths. Canonicalize inputs (resolve `.`, `//`, leading `./`) before comparison.

4. **No file-write concurrency protection** — StateManager, EvolutionEngine, and the agent itself all write to overlapping files without locking. In a real execution environment this risks partial writes.
5. **Evolution doesn't store pre-change state for all rollback paths** — `remove_node` rollback depends on a `node_backup` field in details that nothing in the apply path populates. Rollback of remove_node will silently fail unless the caller remembered to include the backup.

## Immutability guard and evolution safety

The `IMMUTABLE_FILES` frozenset contains three exact strings: `"kernel/BOOT.md"`, `"kernel/constitution.md"`, `"runner.py"`. The `validate_change` method checks `details.get("target_file")`, `details.get("path")`, `details.get("file")`, and `details.get("prompt_file")` against this set using string equality rather than path normalization:

```python
target_file = details.get("target_file", "")
if target_file in IMMUTABLE_FILES:
    return (False, f"Cannot modify protected file: {target_file}")
```

A proposal that passes `"./kernel/BOOT.md"` or `"kernel//BOOT.md"` or `"kernel/./constitution.md"` bypasses the check entirely. The fix: normalize all path values through `os.path.normpath()` before comparison, and normalize the IMMUTABLE_FILES entries at module load.

Additionally, the `modify_prompt` apply path writes to any file under `kernel_dir`:

```python
prompt_path = self.kernel_dir / prompt_file
prompt_path.parent.mkdir(parents=True, exist_ok=True)
with open(prompt_path, "w", encoding="utf-8") as f:
    f.write(new_content)
```

If `prompt_file` is `"../constitution.md"`, the write escapes the prompts directory. The validation catches `details["prompt_file"]` against IMMUTABLE_FILES, but `"../constitution.md"` wouldn't match `"kernel/constitution.md"`. The resolved absolute path should be checked against the resolved absolute paths of protected files.

## Runner loop semantics

The main loop in `runner.py` advances through the graph by always selecting `transitions[0]["to"]` as the next node:

```python
transitions = graph.get_available_transitions(node["id"])
if transitions:
    next_node_id = transitions[0]["to"]
```

No condition evaluation occurs. The `advance()` method on GraphExecutor *does* match conditions, but the runner never calls it. In non-dry-run mode, the kernel always follows the first transition from every node (init -> plan -> code -> test -> review -> reflect -> evolve -> plan -> ...) in a fixed cycle until max_iterations.

The graph.yaml defines conditions like `"plan_ready"`, `"tests_pass"`, `"review_needs_changes"` that imply dynamic evaluation, but the runner has no hook for this. The non-dry-run path produces a state file that claims it traversed nodes it never meaningfully executed. If this is intentional scaffolding, it should be documented; if not, the runner needs a condition-evaluation callback or agent integration point.

## BOOT.md state ownership conflict

Step 7 tells the agent to "Update `kernel/state.yaml` with results" and "Advance `current_node` to the next node." The runner also does this via `state_mgr.set_current_node()` and `state_mgr.save_state()`. Two writers to the same state file produces last-write-wins semantics.

If the kernel is meant to be invoked *by* the runner, the agent shouldn't also write state.yaml. If the kernel is meant to be read directly by an AI agent *outside* the runner (e.g., Cursor reading BOOT.md), then the runner's state management is dead code. BOOT.md should declare which mode it's designed for and drop the conflicting instructions for the other mode.

## Rollback gap in evolution engine

`apply_change` for `remove_node` calls `self.graph_executor.remove_node(node_id)` but doesn't snapshot the removed node's definition into the change record. The rollback path relies on `details.get("node_backup")`:

```python
elif change_type == "remove_node":
    node_dict = details.get("node_backup", details.get("node", {}))
    if node_dict:
        self.graph_executor.add_node(node_dict)
```

If `propose_change` is called with `{"node_id": "code"}` but no `node_backup`, rollback finds an empty dict and silently does nothing. The engine should snapshot the node definition during `apply_change` (before deletion) and store it in the change record for later rollback.

## Disk reads in the runner loop

`is_complete()` reads `progress.yaml` from disk on every call within the runner's tight loop (`for i in range(args.max_iterations)`). With max_iterations=30 this is negligible, but the pattern doesn't scale and could be replaced by checking the in-memory progress state.

## Test gaps

What's not tested: adversarial path inputs to the evolution engine (the normalization bypass), directory traversal via `modify_prompt` with `"../..."` paths, behavior when history.jsonl is corrupted or grows very large, and the dual-writer conflict between runner and agent.
