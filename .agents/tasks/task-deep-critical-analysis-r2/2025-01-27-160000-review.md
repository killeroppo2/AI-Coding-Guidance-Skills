# Seven rounds of kernel hardening: contracts, isolation, tasks, skills, feedback, recovery, and integration tests

Seven features ship together: a formal output format contract with validator, workspace isolation via constitution and path enforcement, a dependency-aware TaskManager with YAML persistence, automatic skill selection by goal-keyword matching, a reflector-to-evolution feedback loop with auto-apply, pre-execution visit tracking with retry strategies (continue/skip/backoff), and integration tests exercising Mode 3 end-to-end via MockAIRunner. The result transforms the kernel from a graph-walking scaffold into a system with defined behavioral contracts, resource boundaries, and closed-loop self-improvement.

**Watch for:** The feedback loop auto-applies evolution proposals above a confidence threshold with no human gate and no rollback budget — a single bad cycle could chain-apply multiple graph mutations before metrics detect regression (confirmed). The workspace isolation boundary is enforced only by prompt instructions (constitution + coder.md), not by code; nothing in runner.py or the evolution engine actually validates that FILES_WRITTEN paths are within workspace_path (confirmed). The `_load_current_task` logic in ContextAssembler duplicates TaskManager's `get_next_task` algorithm without sharing code, creating a drift risk as either implementation evolves (confirmed).

## High-level view

The output format contract (`output_format.md` + `OutputContractValidator`) defines TRANSITION and STATUS as required response lines, validated before state advances. The validator cross-references transition values against graph.yaml, closing the "silent fallback" gap raised in the prior review — contract violations are now logged and the node retries rather than silently advancing.

Workspace isolation adds Article IX to the constitution and modifies coder.md to forbid writes outside `workspace_path`. The boundary is purely advisory: no runtime enforcement prevents an AI or the evolution engine from writing anywhere on disk. This becomes a real concern as the feedback loop gains auto-apply power.

TaskManager provides CRUD over `memory/tasks.yaml` with dependency-aware selection and cycle detection via DFS. ContextAssembler independently reimplements the same task selection logic to inject the current task into prompts — the two implementations are already subtly inconsistent (ContextAssembler checks `in_progress` first; TaskManager doesn't).

The feedback loop runs after every Mode 3 iteration: reflector analyzes → generates proposals → auto-applies all above 0.7 confidence via `EvolutionEngine.apply_if_confident`. No cap on per-cycle applies, no dry-run mode, no way to preview what will change.

Error recovery moves visit tracking to pre-execution and adds three retry strategies. The `--resume` flag now resets `node_visits`, directly addressing the prior review's stale-counts concern. The `skip` strategy advances without a valid AI transition, and `backoff` blocks the main loop with `time.sleep` up to 60s per retry.

Integration tests monkeypatch `subprocess.run` to simulate AI responses per node, exercising full cycles. Not tested: concurrent access to tasks.yaml, feedback loop applying mutations mid-run, `--resume` with pre-existing errors, skill selection against actual skill index files.

## Feedback loop auto-apply without guardrails

The `FeedbackLoop.run_cycle` method calls `self.evolution_engine.apply_if_confident(proposals, self.threshold)` on every iteration:

```python
def apply_if_confident(self, proposals: list, threshold: float = 0.7) -> list[dict]:
    applied: list[dict] = []
    for proposal in proposals:
        confidence = proposal.get("confidence_score", 0.0)
        if confidence <= threshold:
            continue
        change = self.propose_change(change_type, details, reason)
        valid, _ = self.validate_change(change)
        if not valid:
            continue
        success = self.apply_change(change)
        if success:
            applied.append(change)
    return applied
```

The feedback loop runs after *every* successful iteration in Mode 3, meaning the kernel can mutate its own graph and prompts multiple times per run. The `revert_if_worse` mechanism exists but is never invoked automatically — it requires the caller to snapshot metrics, wait, then compare. There's no built-in "apply one, measure, decide" cadence.

A concrete scenario: the code node fails 3 times. The reflector notices the pattern, generates proposals to modify the coder prompt and add a retry node. All score 0.8+. They all apply. The graph now has an untested topology. If the next iteration fails, the feedback loop generates *more* proposals against the already-mutated state. Compound drift with no checkpoint.

## Workspace enforcement is advisory-only

The workspace boundary operates at three levels, all advisory:

1. Constitution Article IX says "AI agent must never write generated project files outside the workspace directory"
2. coder.md says "You may NEVER write to kernel/, memory/, knowledge/"
3. runner.py calls `state_mgr.set_workspace(project_name)` which creates the directory

In Mode 3 execution, after the AI responds with `FILES_WRITTEN: kernel/prompts/evil.md`, the runner parses those paths via `OutputContractValidator._parse_files_written` but never validates them against `workspace_path`. The contract result stores them as data; the runner doesn't check containment.

The evolution engine's `validate_change` checks IMMUTABLE_FILES (BOOT.md, constitution.md, runner.py) but not workspace boundaries. A confident proposal to `modify_prompt` with `prompt_file: "../memory/state.yaml"` would only be caught by the traversal check against `kernel_dir`, not by workspace containment logic.

The combination of "no human gate" + "no runtime path validation" + "auto-apply above confidence" creates a surface for unintended state corruption.

## Duplicated task selection in ContextAssembler

`ContextAssembler._load_current_task` implements:

```python
status_map = {t["id"]: t.get("status", "pending") for t in tasks}
# First look for in_progress
for task in tasks:
    if task.get("status") == "in_progress":
        current = task
        break
# If none, find first eligible pending
pending = [t for t in tasks if t.get("status") == "pending"]
pending.sort(key=lambda t: t["id"])
for task in pending:
    deps = task.get("dependencies", [])
    if all(status_map.get(dep) == "done" for dep in deps):
        current = task
        break
```

`TaskManager.get_next_task` implements the same algorithm minus the `in_progress` check. If TaskManager moves a task to `in_progress`, the assembler shows it to the AI, but on restart, TaskManager's `get_next_task` (which only finds `pending` tasks) will skip the in_progress one and pick the next eligible. The behaviors are already subtly inconsistent — they'll diverge further as selection criteria evolve.

## Skill selector matching failures

```python
goal_words = set(goal.lower().split())
tag_matches = sum(1 for tag in tags if tag.lower() in goal_words)
desc_matches = sum(1 for word in goal_words if word in desc_words)
```

Fails on:
- Plurals/conjugation: "testing" doesn't match tag "test"
- Multi-word tags: tag "rest-api" won't match goal word "rest" (tag compared as a single string)
- Hyphenated goal words: "REST-API" → "rest-api" as one token, won't match tag "rest" or "api" separately

The `max_skills=5` limit means missed matches can push relevant skills below the cutoff entirely.

## Backoff and skip retry behavior

The `skip` strategy advances past a failing node by taking the first transition:

```python
if args.retry_strategy == "skip":
    transitions = graph.get_available_transitions(node["id"])
    if transitions:
        next_node_id = transitions[0]["to"]
        state_mgr.set_current_node(next_node_id)
```

This advances even when the AI hasn't indicated a valid transition, potentially entering the next node without the preconditions that node expects (e.g., skipping `plan` enters `code` without a plan file).

The backoff strategy calls `time.sleep(min(2 ** (visit_count - 1), 60))` inline. After 6 visits this hits the 60s cap and stays there. No timeout on total accumulated backoff, no operator feedback, no signal handling to abort cleanly.
