# Five-feature expansion: Ralph adapter, philosophy heuristics, setup checker, progress reporter, capability assessment

This change adds five new subsystems to the self-evolving kernel: a Ralph integration adapter that converts between the kernel's task format and Ralph's prd.json; a philosophy-to-operations layer that translates dao/strategy principles into concrete decision functions; a first-run setup checker with `--check` flag; a human-readable progress reporter with `--verbose` and `--status` flags; and a capability assessor that measures skill coverage against a goal. Together they extend the kernel from a loop executor into something that can introspect its own readiness, communicate progress, and bridge to an external autonomous agent. The implementation is well-tested (~2000 lines of tests for ~750 lines of production code), and the features integrate at clear seams in runner.py.

**Watch for:** Duplicated logic between `assess_terrain` and `CapabilityAssessor.assess_capabilities` doing nearly identical tokenization/matching with subtly different filtering rules (confirmed). The `should_simplify` call in `reflector.py` passes `count` as both `task_complexity` and `failure_count`, making `task_complexity` semantically meaningless (confirmed). Ralph adapter's `import_from_prd_json` silently produces incorrect dependency chains when stories share the same priority value (confirmed).

## High-level view

The Ralph adapter provides bidirectional conversion between kernel tasks and Ralph's prd.json format, with the export path triggered automatically at end-of-run when `--execution-mode ralph` is set. The import path assumes a strict linear dependency chain derived from priority ordering, which breaks down when priorities collide or the relationship is non-linear.

The philosophy layer introduces four decision functions (`should_stop_iterating`, `should_simplify`, `should_retreat`, `assess_terrain`) wired into the reflector and runner. `assess_terrain` is a near-duplicate of `CapabilityAssessor.assess_capabilities` with different stop-word handling and output structure, creating a maintenance divergence risk.

The setup checker validates Python version, pyyaml presence, kernel file existence, graph loadability, and skill path resolution. It's gated behind `--check` and also callable standalone.

The capability assessor tokenizes goals, matches against skill tags/descriptions, writes `assessment.yaml`, and prints warnings for low coverage. Its tokenization differs from `assess_terrain`'s by applying a stop-word list and a minimum-length filter, meaning the two will disagree on coverage for the same goal+skills inputs.

## Duplicated goal-skill matching logic

`kernel/philosophy/principles.py:assess_terrain` and `kernel/capability_assessment.py:CapabilityAssessor.assess_capabilities` both tokenize a goal string, match tokens against skill tags and descriptions, and return coverage metrics. They differ in:

- **Stop words**: `assess_terrain` only filters words by length (`> 2`). `CapabilityAssessor` applies a 30-word stop list *and* a `>= 3` length filter.
- **Output shape**: `assess_terrain` returns `coverage_score` + `recommendation`; `CapabilityAssessor` returns `confidence` + `gaps` + `covered` skill names.
- **Integration point**: `assess_terrain` is unused in the runner (only called from tests). `CapabilityAssessor` is used in runner.py.

For the same goal "Build a REST API with auth" and the same skills, these two functions will report different coverage numbers because `assess_terrain` keeps "build" and "with" (length > 2) while `CapabilityAssessor` strips them as stop words. If `assess_terrain` is later wired into the runner (its docstring suggests it should be), its recommendation thresholds (0.7/0.4) will fire at different actual coverage levels than the assessor's warning thresholds (0.3/0.7).

## should_simplify semantics vs. usage

In `kernel/philosophy/principles.py`, `should_simplify(task_complexity, failure_count)` is documented as: "Returns True if failure_count >= 3, suggesting the task should be split. task_complexity is number of steps/subtasks." The `task_complexity` parameter is never used in the function body.

In `kernel/reflector.py:173`, the call is:
```python
if should_simplify(count, count):
```

where `count` is the failure count for a node. Passing the same value as both arguments signals that the caller doesn't have access to actual task complexity. Either the parameter should be removed, or the reflector should pass meaningful subtask data.

## Ralph adapter dependency chain assumption

`import_from_prd_json` sorts stories by priority and assigns `T-{i:03d}` IDs sequentially. Each story with priority > 1 gets a dependency on the previous task in the sorted order:

```python
if story.get("priority", 1) > 1 and i > 0:
    dependencies = [f"T-{i:03d}"]
```

This creates a linear chain. If two stories share the same priority (e.g., both are priority 2), their relative order after sorting is undefined (Python's sort is stable, but the input from a JSON file has no guaranteed order for equal-priority items). The dependency chain will be silently wrong: one becomes a prerequisite of the other despite being conceptually parallel.

## progress_history stall detection is inert

`should_stop_iterating` checks:
```python
progress_history = state.get("progress_history", [])
if len(progress_history) >= 5:
    last_five = progress_history[-5:]
    if len(set(last_five)) == 1:
        return True
```

No code path in this diff (nor in the runner loop) ever appends entries to `progress_history`. The field remains empty during execution, so this stall-detection branch can never trigger. The error-repetition branch of `should_stop_iterating` (checking reflections for repeated issues) does work because `feedback_loop.run_cycle` writes to `reflections.jsonl`, but the progress-based path is dead.

## Capability assessment overwrites on resume

`CapabilityAssessor.write_assessment` creates `memory/assessment.yaml` unconditionally on every non-dry-run invocation. On `--resume`, the prior assessment is silently overwritten even though the goal hasn't changed. The assessment timestamp will always reflect the last run start rather than when the goal was actually assessed, which could mislead downstream tooling that reads `assessment.yaml` for historical tracking.
