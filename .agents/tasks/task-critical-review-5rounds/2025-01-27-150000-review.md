# Five rounds of critical fixes for the self-evolving kernel

Five features land in one branch: Mode 3 subprocess-based AI execution in the runner, dependency-aware skill composition with topological ordering and token budgets, retry tracking with stuck detection and handler redirection, evolution metrics with auto-rollback, and boot resilience with consolidated prompt generation. The result is a system that can invoke a real AI, detect when it's spinning, measure whether evolution changes help or hurt, and generate a single-shot prompt from all kernel components.

**Watch for:** The runner's Mode 3 subprocess invocation passes AI output through `_parse_transition` which only matches exact condition strings from graph.yaml, making it brittle to minor AI wording variations (confirmed). `ContextAssembler` and `BootstrapGenerator` are near-duplicates that overlap significantly in purpose and will drift apart (confirmed). The stuck detection resets visit counts only on fresh starts (not on `--resume`), meaning a resumed session inherits stale visit history that may trigger false stuck detection (likely).

## High-level view

Mode 3 execution pipes assembled context to an AI CLI tool as stdin and parses stdout for a `TRANSITION: <condition>` line. If no match is found, it falls back to the first transition (same as Mode 1). The transition parsing is rigid: the AI must output an exact condition string from graph.yaml, with no fuzzy matching or structured output format. The fallback is silent, making debugging impossible when the AI produces slightly wrong output.

Skill composition gained real substance: topological ordering via Kahn's algorithm, conflict detection via `composable_with` metadata, phase-based auto-selection, and token budget enforcement by dropping lowest-priority skills. The cycle detection gracefully degrades to original ordering on cycles.

Stuck detection introduces `node_visits` tracking and a `check_stuck` method comparing visit counts against per-node `max_retries`. When stuck, the runner redirects to a `stuck_handler` node if one exists, otherwise terminates. The interaction with `--resume` is problematic: stale visit counts from previous sessions carry over and may trigger false stuck detection.

The evolution subsystem gained `EvolutionMetrics` for sliding-window performance tracking and `revert_if_worse` for automatic rollback. Metrics are in-memory only with no persistence, limiting their utility to single sessions. The reflector now categorizes failures and attaches confidence scores to proposals.

`BootstrapGenerator` consolidates all kernel files into one prompt for `--generate-prompt`, while `ContextAssembler` handles Mode 3 per-iteration prompts. The two produce different formatting and content for the same conceptual operation, creating a debugging trap when operators preview prompts.

1. **Transition parsing brittleness** — `_parse_transition` requires an AI to output the exact string `TRANSITION: tests_pass` matching a graph.yaml condition. No normalization, no partial matching. Any deviation falls back to first-transition scaffolding, silently losing the AI's intent. Consider accepting structured JSON output or normalizing whitespace/casing. (confirmed)
2. **ContextAssembler/BootstrapGenerator duplication** — Two classes assemble nearly identical context from the same files with slightly different formatting (`=== BOOT SEQUENCE ===` vs `--- BOOT SEQUENCE ---`). `--generate-prompt` uses BootstrapGenerator, Mode 3 uses ContextAssembler. They will drift. Consolidate or make one delegate to the other. (confirmed)
3. **Stale visit counts on resume** — `--resume` loads existing state including `node_visits`, but these counts reflect a previous session's traversal. A node visited 4 times previously starts at 4 in the resumed session, potentially triggering stuck detection immediately if `max_retries` is 5. Reset `node_visits` on resume or count only visits since the resume point. (likely)
4. **Subprocess command splitting via str.split()** — `args.ai_command.split()` splits on whitespace, which breaks commands containing quoted arguments like `claude --print --model "claude-3"`. Use `shlex.split()` instead. (confirmed)
5. **Silent fallback masks AI failures** — When Mode 3 gets AI output but finds no TRANSITION line, it silently takes the first transition. No logging, no warning, no record that the AI's response was unparseable. The system proceeds as if the AI chose the first transition, making debugging impossible. (confirmed)
6. **Token budget estimation** — `len(section) // 4` as a token estimator will significantly undercount for code-heavy skills (tokens/char closer to 1:3) and overcount for prose. The truncation message may confuse users who expect accurate counts. (possible)
7. **No timeout on BootstrapGenerator file reads** — `BootstrapGenerator.generate()` reads multiple files synchronously with no timeout protection. If a path hangs, the entire call blocks indefinitely. Lower priority for CLI, but relevant for Mode 3 where it could run per-iteration. (possible)
8. **Metrics are in-memory only** — `EvolutionMetrics` stores all data in a `deque` with no persistence. Restarting the runner loses all performance history, making `revert_if_worse` decisions unreliable across sessions. (confirmed)

## Mode 3 execution and transition protocol

The runner's Mode 3 path assembles context via `ContextAssembler`, pipes it to a subprocess as stdin, and scans stdout for a `TRANSITION:` line:

```python
result = subprocess.run(
    args.ai_command.split(),
    input=context_prompt,
    capture_output=True,
    text=True,
    timeout=args.timeout,
)
ai_output = result.stdout
transition_condition = _parse_transition(ai_output)
```

Two problems compound here. First, `args.ai_command.split()` uses naive whitespace splitting. A command like `claude --print --system "Be concise"` becomes `["claude", "--print", "--system", "\"Be", "concise\""]` which will fail or produce wrong behavior. `shlex.split()` handles quoting correctly.

Second, `_parse_transition` scans for an exact `TRANSITION: <condition>` line. The returned string must match a condition in graph.yaml verbatim. When no match is found among the current node's transitions, the runner falls through to `transitions[0]["to"]` with no indication that the AI's intent was lost. No log message, no error recorded in state, no metric tracked. Mode 3 degrades to Mode 1 whenever the AI produces slightly wrong output, with no signal to the operator.

The subprocess also ignores `result.returncode` entirely. A non-zero exit from the AI command is not treated as an error; only `FileNotFoundError` and `TimeoutExpired` are caught. An AI tool that exits with code 1 and stderr output gets its stdout parsed normally.

## Dual context assemblers

`ContextAssembler.assemble()` produces sections with `=== SECTION NAME ===` headers. `BootstrapGenerator.generate()` produces sections with `--- SECTION NAME ---` headers. Both read BOOT.md, state.yaml, the current node's prompt, dao.md, strategy.md, and loaded skills. BootstrapGenerator adds constitution as a separate section; ContextAssembler does not.

The runner uses BootstrapGenerator for `--generate-prompt` and ContextAssembler for Mode 3. The prompt an operator previews with `--generate-prompt | claude --print` has different formatting and content (includes constitution separately) than what the AI receives during `--ai-command "claude --print"`. Testing the prompt pipeline produces behavior that diverges from actual execution.

## Stuck detection and resume interaction

`check_stuck(max_retries_map)` iterates all visited nodes and returns True for the first one exceeding its limit. On `--resume`, existing state is loaded including `node_visits` from the previous session. A node visited 3 times previously starts the resumed session at 3. If max_retries is 5, the node only gets 2 more visits before stuck triggers. There's no concept of "visits this session" vs "visits lifetime," making `--resume` after a partial success unpredictable.

## Evolution metrics without persistence or integration

`EvolutionMetrics` uses `collections.deque(maxlen=window_size)` for per-node storage, entirely in memory. The runner never instantiates `EvolutionMetrics` — it's available for the evolution engine and reflector to use, but the current runner loop doesn't call `record_iteration`. Metrics only accumulate if external code explicitly records them.

`revert_if_worse` is a building block, but nothing in the current codebase automates the "measure before, apply, measure after, decide" cycle. It requires the caller to snapshot metrics, wait for iterations, then call the comparison. Without persistence, this workflow can't span runner restarts.
